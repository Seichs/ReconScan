SQL Injection

CRITICAL

URL: https://vulnerable-app.com/login

Parameter: username

OWASP Category: A03:2021 ‚Äì Injection

CWE: CWE-89, CWE-564, CWE-943

üéì What is this vulnerability?

SQL injection occurs when user input is incorrectly filtered or insufficiently sanitized, allowing attackers to execute arbitrary SQL commands on the database.

Technical Explanation
SQL injection vulnerabilities arise when applications dynamically construct SQL queries using user-supplied input without proper validation or parameterization. Attackers can manipulate these inputs to alter the intended SQL logic, potentially gaining unauthorized access to data, modifying database contents, or executing administrative operations. The vulnerability stems from the fundamental confusion between code and data in SQL queries. When user input is directly concatenated into SQL strings, the database engine cannot distinguish between the intended query structure and malicious input that changes the query's logic.

üíº Business Impact

SQL injection represents one of the most severe security vulnerabilities, with potential impacts including: ‚Ä¢ **Data Breach**: Complete database compromise exposing sensitive customer information, financial records, and proprietary data ‚Ä¢ **Financial Loss**: Direct monetary theft, regulatory fines (GDPR, CCPA), legal costs, and reputation damage ‚Ä¢ **Business Disruption**: System downtime, data corruption, and loss of customer trust ‚Ä¢ **Compliance Violations**: Failure to meet regulatory requirements (PCI DSS, HIPAA, SOX) ‚Ä¢ **Competitive Disadvantage**: Loss of intellectual property and trade secrets

üõ†Ô∏è How to Fix This

  1. Implement Parameterized Queries: Use prepared statements or parameterized queries to separate SQL logic from data
  2. Input Validation: Validate and sanitize all user inputs before processing
  3. Principle of Least Privilege: Configure database user with minimal required permissions
  4. Error Handling: Implement proper error handling to prevent information disclosure

OWASP Category: A03:2021 ‚Äì Injection

CWE: CWE-77, CWE-78, CWE-88

üéì What is this vulnerability?

Command injection occurs when applications execute system commands using unvalidated user input, allowing attackers to execute arbitrary commands on the host operating system.

Technical Explanation
Command injection vulnerabilities arise when applications construct system commands using user-supplied input without proper validation or sanitization. This allows attackers to inject additional commands or modify the intended command execution, potentially gaining full control over the host system. The vulnerability typically occurs when applications use functions like system(), exec(), or shell_exec() with user-controlled data. Command separators (;, &&, ||, |) and command substitution operators (`, $()) can be used to chain additional commands or execute arbitrary code within the same execution context.

üíº Business Impact

Command injection represents a critical security risk with severe potential impacts: ‚Ä¢ **Complete System Compromise**: Full control over the host operating system and infrastructure ‚Ä¢ **Data Exfiltration**: Access to all files and databases on the compromised system ‚Ä¢ **Lateral Movement**: Using compromised systems as launching points for further attacks ‚Ä¢ **Service Disruption**: System downtime, data destruction, and operational impact ‚Ä¢ **Compliance Violations**: Breach of security controls and regulatory requirements ‚Ä¢ **Intellectual Property Theft**: Access to source code, algorithms, and trade secrets

üõ†Ô∏è How to Fix This

  1. Avoid System Commands: Use safe APIs instead of executing system commands
  2. Input Sanitization: Whitelist allowed characters and escape shell metacharacters
  3. Subprocess Security: Use secure subprocess execution with proper argument handling

OWASP Category: A01:2021 ‚Äì Broken Access Control

CWE: CWE-22, CWE-23, CWE-36

üéì What is this vulnerability?

Directory traversal vulnerabilities allow attackers to access files and directories outside the intended application scope by manipulating file path references.

Technical Explanation
Directory traversal (path traversal) vulnerabilities occur when applications use user-supplied input to construct file paths without proper validation. Attackers can use special characters like '../' (dot-dot-slash) to navigate outside the intended directory structure and access sensitive files on the server. This vulnerability commonly affects file upload functionality, document viewers, and any feature that processes file paths. The attack exploits the way operating systems resolve relative paths, allowing traversal to parent directories and access to system files, configuration files, and other sensitive resources.

üíº Business Impact

Directory traversal vulnerabilities can lead to significant security breaches: ‚Ä¢ **Sensitive File Exposure**: Access to configuration files, passwords, and system information ‚Ä¢ **Source Code Disclosure**: Exposure of application logic and potential additional vulnerabilities ‚Ä¢ **System Information Leakage**: Details about server configuration and infrastructure ‚Ä¢ **Credential Theft**: Access to password files and authentication tokens ‚Ä¢ **Compliance Violations**: Unauthorized access to regulated data and information ‚Ä¢ **Intellectual Property Loss**: Exposure of proprietary algorithms and business logic

üõ†Ô∏è How to Fix This

  1. Path Validation: Validate and canonicalize file paths before processing
  2. Whitelist Approach: Use whitelist of allowed files/directories instead of blacklist
  3. Chroot Jail: Implement chroot jail to restrict file system access

üéì What is this vulnerability?

Security headers are HTTP response headers that help protect web applications from various attacks by instructing browsers how to handle content and enforce security policies.

Technical Explanation
Security headers provide defense-in-depth protection by leveraging browser security features. When applications fail to implement proper security headers, they miss critical opportunities to protect users from various attacks including XSS, clickjacking, MITM attacks, and information disclosure. Key security headers include: ‚Ä¢ Content-Security-Policy (CSP): Prevents XSS and data injection attacks ‚Ä¢ X-Frame-Options: Protects against clickjacking attacks ‚Ä¢ Strict-Transport-Security (HSTS): Enforces HTTPS connections ‚Ä¢ X-Content-Type-Options: Prevents MIME type sniffing attacks ‚Ä¢ Referrer-Policy: Controls referrer information leakage

üíº Business Impact

Missing security headers expose applications to various risks: ‚Ä¢ **Increased Attack Surface**: Vulnerability to clickjacking, XSS, and injection attacks ‚Ä¢ **Data Leakage**: Uncontrolled referrer information and content type sniffing ‚Ä¢ **Man-in-the-Middle Attacks**: Lack of transport security enforcement ‚Ä¢ **Compliance Issues**: Failure to implement security best practices ‚Ä¢ **Brand Damage**: Security incidents due to preventable vulnerabilities ‚Ä¢ **User Trust**: Reduced confidence in application security

üõ†Ô∏è How to Fix This

  1. Implement Security Headers: Add missing security headers to HTTP responses
  2. Configure CSP: Set up Content Security Policy appropriate for your application
  3. Enable HSTS: Configure HTTP Strict Transport Security for HTTPS enforcement
